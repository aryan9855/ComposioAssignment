Headline thesis
OAuth2 + API key win the landscape.

Most of the catalog is buildable with public docs and standard authentication flows. The biggest friction is not API breadth — it is who can actually get access and whether the workflow is partner-gated.

Easy wins: CRM, docs, support, dev tools
Careful path: ads, finance, research APIs

Headline findings

  • Auth is concentrated: OAuth2, API key, and token-based flows dominate the catalog. Basic auth is a minority but still appears in older SaaS surfaces.
  • Self-serve is common in SaaS: CRM, helpdesk, messaging, project tools, docs, and ecommerce tooling are typically self-serve or developer-trialable.
  • Governance is the main blocker: finance, ad platforms, enterprise connectors, and data-intellectual-property apps are more often gated by admin approval, paid plans, partner programs, or compliance review.
  • Best buildability wins: products with public REST/GraphQL docs, documented sandbox credentials, and stable entity models are the easiest agent-toolkit candidates.

The full design is intentionally transparent: the page shows both the machine-generated classification and where human judgment is still required.

What the app / agent actually does

Built artifact

A lightweight deterministic Python agent that takes the 100-app list, clusters it, infers auth/access posture from the public docs hint, and writes a structured JSON output.

Input → app + docs hint
Output → auth + access + buildability
Artifact → research_output.json

Where a human was needed

  • Confirm whether a provider is truly self-serve or partner-gated
  • Resolve ambiguous docs for ads, finance, and research APIs
  • Check whether public docs reflect current access policy or old posture
  • Label confidence when the docs are incomplete

Honesty and limitations

This is a heuristic research agent, not a compliance-grade legal or enterprise-access system. It is designed to classify patterns from public docs, but it must be explicit when evidence is incomplete.

  • What it can do well: infer auth pattern, public API shape, and likely self-serve posture from docs hints.
  • What it cannot safely claim: a definitive access policy, a guaranteed partner status, or a production-ready toolchain without verification.
  • What it does on uncertain apps: labels the verdict as medium-confidence and shows the reason instead of pretending certainty.

Why this is trustworthy

The page intentionally shows the process, not just the final matrix. The agent outputs evidence URLs, produces category-level patterns, and labels confidence where public documentation is ambiguous. That means the reviewer can see both the first-pass heuristics and the human verification steps.
